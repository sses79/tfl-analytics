TfL Analytics Hosting Architecture

Angular dashboard, Web API and Azure SignalR

The user interface and API are deployed independently so each can use the Azure hosting model that fits it best. Angular is a globally served static application, ASP.NET Core runs as a scale-to-zero container, and Azure SignalR adds low-latency server push without making the browser repeatedly poll the API.

How the application is hosted in Azure

Static content, API compute and persistent real-time connections have different operational needs, so they are hosted as separate Azure resources.

Three hosting roles
Frontend hosting

Azure Static Web Apps Free

Hosts the compiled Angular browser assets in West Europe and serves the four lazy-loaded routes.

  • Production bundle deployed with Static Web Apps CLI.
  • SPA fallback rewrites route requests to index.html.
  • CSP restricts scripts, frames and API connection destinations.
  • No Angular server process is required in Azure.
API compute

Azure Container Apps Consumption

Runs the ASP.NET Core 10 image from private Azure Container Registry with public HTTPS ingress on container port 8080.

  • 0.25 vCPU and 0.5 GiB memory per replica.
  • Scales from zero to two replicas.
  • HTTP scaling target: 20 concurrent requests.
  • Liveness probe calls /health/live.
Connection service

Azure SignalR Service Free F1

Holds browser connections and fans out messages published by the API or Processing Functions.

  • Default service mode with one Free F1 unit.
  • Local key authentication disabled.
  • API and processing identities receive App Server roles.
  • Dashboard origin is the only configured browser origin.

Deployment topology

Angular and the API have separate hostnames. The browser downloads static assets from Static Web Apps, then calls the Container App and establishes a SignalR connection.

How Azure SignalR improves the Angular dashboard

REST remains the source of a complete snapshot. SignalR complements it with incremental updates, reducing stale screens and avoiding a separate polling loop on every open dashboard.

Server push
  1. Angular SignalRService → Connects to /hubs/dashboard using the Microsoft SignalR client and enables automatic reconnect.
  2. Container App API → ASP.NET Core Azure SignalR integration negotiates the client connection using the API user-assigned managed identity.
  3. Azure SignalR Service → Holds the persistent client connection independently of Container App replica lifetime and scale-to-zero cycles.
  4. Processing Functions → After persistence, publish arrivalsUpdated, lineStatusChanged, or alertRaised through Azure SignalR's REST endpoint using managed identity.
  5. Angular signals and effects → Update the relevant page immediately or trigger a focused API refresh without reloading the application.
Lower latency New arrivals, disruptions and alerts appear shortly after processing.
Less browser polling One connection replaces frequent per-page HTTP refresh loops.
Independent scaling SignalR retains client connections while API replicas scale separately.
Resilient UX Automatic reconnect plus manual refresh keeps the dashboard recoverable.
SignalR messages are transient notifications, not the system of record. A newly opened or reconnected page still queries the Web API so Cosmos DB and Azure SQL remain the authoritative state.

Security, scaling and operational boundaries

The browser is intentionally separated from Azure data services. It receives public application data only through the API and SignalR contracts.

Managed identity first

Container image delivery

The API image is stored in private Basic ACR. The Container App's user-assigned identity has AcrPull; ACR admin credentials are disabled.

Data-plane access

The API identity queries Cosmos DB and Azure SQL using Azure credentials and RBAC/database grants rather than browser-visible connection strings.

SignalR publishing

API and Processing Function identities have the SignalR App Server role. Local key authentication is disabled and AAD authentication remains enabled.

Browser boundaries

API CORS allows the exact Static Web App origin. Static Web Apps CSP restricts scripts and connect-src to the API hostname.

Cost-conscious API scaling

Container Apps Consumption scales to zero when idle and caps at two replicas. The first request after idle may experience a cold start.

Service health and telemetry

Container Apps probes /health/live. Application Insights captures API requests and dependencies; SignalR connectivity logs are available through diagnostics.
