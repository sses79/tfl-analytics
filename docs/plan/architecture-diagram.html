Live event: arrival prediction

A station arrival prediction is archived unchanged, normalized for fast queries, evaluated for prediction slippage, and pushed to the live dashboard.

ArrivalPredictionObserved
Function PollArrivals Creates a deterministic versioned envelope. EventEnvelope<ArrivalPredictionObserved>
Messaging Event Hubs Publishes to tfl-events.
Function ArchiveEventHubEvents Archives raw JSON and queues a lightweight pointer.
Raw storage Blob / Data Lake raw Gzip JSON partitioned by date and station. eventType=arrival/.../stationId=...
Queue + Function processing → ProcessQueuedEvent Validates, deduplicates and normalizes.
Operational store Cosmos DB: live-events Partition key /stationId; seven-day TTL.
Real time Azure SignalR Broadcasts the browser update. arrivalsUpdated

Alert branch: a prediction slipping by more than 120 seconds creates an ArrivalPredictionSlippage alert candidate.

Line status

A line observation follows the same archive and queue backbone but lands in a line-partitioned Cosmos container and can trigger a disruption alert when status worsens.

LineStatusObserved
Function PollLineStatus Wraps the current status in a versioned envelope. EventEnvelope<LineStatusObserved>
Messaging Event Hubs Publishes to tfl-events.
Function ArchiveEventHubEvents Archives the raw event and queues processing.
Raw storage Blob / Data Lake raw Gzip JSON partitioned by date and line. eventType=line-status/.../lineId=...
Queue + Function processing → ProcessQueuedEvent Compares the previous line observation.
Operational store Cosmos DB: line-status Partition key /lineId; seven-day TTL.
Real time Azure SignalR Broadcasts the status change. lineStatusChanged

Alert branch: a good-to-disrupted transition creates a LineStatusDisruption alert candidate.

Alert workflow

Qualifying arrival or line-status transitions start an idempotent Durable Functions orchestration with retried activities.

AlertCandidate → AlertRaised
Rule output AlertCandidate Deterministic alert ID and source event ID. ArrivalPredictionSlippage | LineStatusDisruption
Orchestration AlertOrchestration Durable instance ID equals the alert ID.
Activity PersistAlert Idempotent insert; duplicate keys are accepted safely.
System of record Table Storage: alerts Alert history keyed by a deterministic row key. SourceEventId, RuleType, DetectedAtUtc
Activity BroadcastAlert Publishes the completed alert outcome. AlertRaised
Real time Azure SignalR Pushes the alert to connected clients. alertRaised

Alert audit

The Durable workflow writes a separate low-cost audit entity after alert persistence, preserving the workflow action and source linkage.

Action: AlertRaised
Orchestration AlertOrchestration Continues after the alert persistence activity.
Activity WriteAlertAudit Upserts an audit entity using the alert identity.
Audit store Table Storage: audit Partition key is the rule type; row key is the alert ID. Action=AlertRaised, SourceEventId
Activity Mock notification Logs the notification outcome for the portfolio workflow.

API and dashboard read paths

The write flows above create the stores. The Web API queries those stores, while SignalR supplies low-latency updates between refreshes.

Cosmos DB: live-events GET /api/stations/{stationId}/arrivals → Arrivals page
Cosmos DB: line-status GET /api/lines/status → Line status page
Table Storage: alerts GET /api/alerts → Alerts page
Combined operational state GET /api/dashboard/summary → Dashboard page
Source of truth: Plan.md, event contracts in TflAnalytics.Contracts, and the current Infrastructure and Processing Functions implementations. Application Insights and Log Analytics receive telemetry across the Functions and API but are omitted from individual lanes to keep the primary data paths readable.
